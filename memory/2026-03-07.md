## [Bot稳定性] /img 长提示词链路仍不稳定
- **结果**：已连续修复 4 类问题：子进程超时缩短到 10 分钟、定期清理僵尸子进程、stdout/stderr 改为线程异步读取避免死锁、JSON 结果输出加 `__RESULT_START__`/`__RESULT_END__` 标记避免被调试日志污染。
- **结果**：另外修复了 Windows `UnicodeEncodeError: 'gbk' codec can't encode character '\u2022'`，在 `cmd_handler.py` 强制 `sys.stdout/sys.stderr` 使用 UTF-8。
- **关键决策**：自动“智能压缩”虽然已接入 Qwen3，但在真实超长 `/img` 提示词下仍可能导致子进程无输出或链路异常；当前更稳的方向是放弃 LLM 压缩，改为“翻译后超长直接截断 + 明确提示用户已自动截断”。
- **待办**：下次继续时，优先把 `/img` 过长提示词策略降级为纯截断，不再调用 Ollama 压缩；先保稳定出图，再谈智能压缩。

## [最终修复] Telegram Bot 的 /img 与 /md 回传链路稳定
- **结果**：`/img` 与 `/md` 都已验证成功，表现为：ComfyUI 正常出图，Telegram Bot 也能正常收图并回传，不再出现 `Error: no output` 或长期卡在 `Generating`。
- **关键决策**：根因不是 ComfyUI，也不是单个命令工作流，而是旧架构依赖 `python -c` 子进程 + stdout/stderr + JSON 标记做结果回传；这条 IPC 链路在 Windows 上极不稳定。
- **关键决策**：最终采用根治方案：`comfy_bot.py` 主进程内直接 `import cmd_handler` 并调用 `cmd_handler.handle()`，彻底移除子进程 JSON 回传。
- **关键决策**：为保证长 `/img` 稳定性，策略改为“原文先截断（500）→ 再翻译 → 英文再截断（380）”，放弃 AI 智能压缩优先路线。
- **待办**：后续如再遇到“已出图但 Bot 不回图”，优先检查是否有人重新引入子进程 stdout/stderr 回传方案；不要再回退到 `python -c` IPC 架构。
